프로그램과 개발 흐름 이해하기

SECTION 01

프로그램이란 무엇인가

프로그램은 사용자의 입력을 받아, 정해진 방식으로 처리하고, 그 결과를 출력물로 보여주는 것입니다.

예를 들어 엑셀 파일을 입력값으로 하여 정해진 규칙에 따라 데이터를 정리하고, 최종 결과를 PDF 파일로 생성하는 과정도 하나의 프로그램입니다.

1
입력 (Input)
사용자가 넣는 자료. 엑셀 파일, 텍스트, 클릭 같은 행동 자체도 입력이 될 수 있습니다.
↓
2
처리 (Process)
정해진 규칙과 순서에 따라 입력을 계산하거나 가공하는 단계. 프로그램의 핵심 로직이 여기 있습니다.
↓
3
출력 (Output)
처리 결과를 사람이 확인할 수 있는 형태로 보여주는 단계. 화면, 파일, 알림 등 다양한 형태가 있습니다.
SECTION 02

개발은 어떤 흐름으로 진행되는가

프로그램을 만드는 과정은 아래와 같은 흐름을 따릅니다.

단계내용
요구사항 확인무엇을 만들지 먼저 정합니다.
"엑셀 자료를 불러와 월간 보고서를 자동으로 만들고 싶다"처럼 목적과 필요한 결과물을 정리합니다.
설계입력자료, 처리 방식, 출력 형태를 미리 정합니다.
어떤 파일을 사용할지, 어떤 기준으로 계산할지, 결과를 어떤 화면이나 파일로 보여줄지 정합니다.
제작정리한 내용을 바탕으로 실제 프로그램을 만듭니다.
AI Agent를 활용하면 코드 작성과 파일 수정 과정의 많은 부분을 도움받을 수 있습니다.
테스트만든 프로그램이 의도한 대로 동작하는지 확인합니다.
오류가 나거나 결과가 다르면 원인을 찾아 수정합니다.
배포다른 사람도 사용할 수 있도록 실행 가능한 형태로 정리합니다.
웹 링크, 실행 파일, 공유 폴더 등 사용 방식에 맞게 제공합니다.
유지보수실제 사용 중 발견되는 오류를 고치고, 업무 기준이 바뀌면 기능을 수정하거나 보완합니다.
AI로 개발해도 사라지지 않는 것

AI가 코드를 대신 작성해준다고 해서 이 흐름 자체가 없어지는 것은 아닙니다.
무엇을 만들지 정하고, 결과를 검토하고, 방향을 판단하는 역할은 여전히 사람의 몫입니다.

SECTION 03

프로그래밍 언어와 자연어, 무엇이 다른가

사람끼리 쓰는 말(자연어)은 조금 모호해도 문맥으로 알아듣습니다. "대충 정리해줘"라고 해도 상대방은 상황에 맞게 판단해서 처리합니다.

하지만 컴퓨터에게 지시하는 프로그래밍 언어는 다릅니다. 정해진 문법과 순서를 정확히 지켜야만 컴퓨터가 이해합니다. 쉼표 하나, 괄호 하나가 빠져도 실행되지 않을 수 있습니다.

언어·도구설명
HTML화면 구조를 짜는 언어
CSS화면의 색상, 배치, 글꼴 같은 디자인을 입히는 언어
JavaScript화면에 클릭, 입력 같은 동작을 만들어주는 언어
Python데이터 처리나 자동화에 많이 쓰이는 언어
참고

"이런 언어들로 화면과 기능이 만들어진다" 정도만 알아두어도 충분합니다.

AI Agent 방식 이해하기

SECTION 01

대화형 AI와 AI Agent, 무엇이 다른가

지금까지 사용해 온 Claude, ChatGPT, Gemini 같은 AI는 주로 사용자의 질문에 답변하거나, 문서 요약·초안 작성·아이디어 정리처럼 하나의 요청에 대한 결과를 제공하는 방식으로 활용되어 왔습니다.

AI Agent도 사용자가 원하는 목표를 설명하고 작업을 맡긴다는 점은 비슷합니다.

다만 차이는 답변을 제공하는 데서 끝나는지, 작업 흐름을 이어가며 결과물까지 만들어내는지에 있습니다.

예를 들어 보고서 PDF를 만들고 싶다면, 대화형 AI는 보고서 구성, 문구, 표 정리 방법을 제안할 수 있습니다. 이후 엑셀 파일을 열고, 내용을 옮기고, 문서를 편집하고, PDF로 저장하는 작업은 사용자가 직접 수행해야 합니다.

반면 AI Agent 방식은 사용자가 목표와 작업 폴더를 지정하면, 필요한 파일을 확인하고 데이터를 정리한 뒤 보고서 파일을 생성하고 PDF 형식으로 저장하는 과정까지 이어갈 수 있습니다.

예시: 보고서 PDF 만들기

작업 폴더에 엑셀 실적 자료와 기존 보고서 양식이 있다고 가정해 보겠습니다.

대화형 AI에 요청하면
방법을 제안받고
나머지는 사람이 처리
  1. 엑셀 자료를 분석해 보고서 문구와 표 구성 방법을 제안받음
  2. 사람이 직접 엑셀과 문서를 열어 내용을 반영
  3. 사람이 최종적으로 PDF 파일로 저장
AI Agent에 요청하면
AI가 작업 흐름을 이어가며
결과물까지 생성
  1. 작업 폴더의 엑셀 자료와 보고서 양식을 확인
  2. 필요한 데이터를 정리하고 보고서 내용을 작성
  3. 보고서 파일을 생성하고 PDF 형식으로 저장
SECTION 02

승인 요청, 이런 뜻입니다

AI 개발 환경에서 AI Agent에게 작업을 시키다 보면 영어로 된 승인 팝업이 뜰 수 있습니다. 내용을 모른 채 무조건 Yes를 누르는 것은 위험할 수 있습니다. AI가 파일을 삭제하거나, 외부 패키지를 설치하거나, 터미널 명령어를 실행하려는 순간일 수 있기 때문입니다.

자주 뜨는 팝업의 의미는 다음과 같습니다.

화면에 표시되는 문구실제 의미판단 기준
Do you want to create this file? AI가 새 파일을 만들려고 합니다 어떤 파일인지 이름을 확인하고 승인
Do you want to edit this file? AI가 기존 파일을 수정하려고 합니다 어떤 파일인지 확인 후 승인
Do you want to delete this file? AI가 파일을 삭제하려고 합니다 삭제 전 반드시 백업 확인

바이브코딩이란 무엇인가

SECTION 01

바이브코딩의 개념

바이브코딩(Vibe Coding)은 코드를 한 줄씩 직접 작성하는 대신, 만들고 싶은 기능을 자연어로 설명하고 AI와 함께 개발을 이어가는 방식을 말합니다.

"엑셀 파일을 업로드하면 자동으로 요약 보고서를 만들어주는 프로그램을 만들어줘"처럼 목적을 말로 전달하면, AI Agent가 필요한 파일을 만들고 수정하며, 실제로 동작하는 프로그램에 가까워지도록 작업을 이어갑니다.

SECTION 02

사람과 AI의 역할 분담

바이브코딩이라고 해서 사람이 할 일이 없어지는 것은 아닙니다. 오히려 무엇을 만들지 정하고 결과를 판단하는 역할이 더 중요해집니다.

사람
방향을 정하고
결과를 검토
  • 무엇을, 왜 만드는지 목적 정의
  • 업무 기준과 조건 설명
  • 결과물이 의도대로 나왔는지 검토
  • 수정이 필요한 방향 판단
AI
코드를 작성하고
결과를 이어감
  • 요청 내용을 바탕으로 코드 작성
  • 필요한 파일 수정
  • 오류 원인 분석
  • 실행 결과를 보고 개선안 제안
SECTION 03

바이브코딩에 대한 오해 정리

바이브코딩을 처음 접하면 "이제 개발을 몰라도 되는구나"라고 생각하기 쉽지만, 정확히는 다릅니다.

오해와 사실
몰라도 된다 vs 알수록 잘 시킨다
바이브코딩은 "개발 지식이 전혀 없어도 완벽하게 된다"는 뜻이 아니라, "개발 개념을 알고 있을수록 AI에게 더 정확하게 요청하고, 결과를 더 잘 판단할 수 있다"는 뜻에 가깝습니다.

GitHub 이해하기

SECTION 01

GitHub의 개념

GitHub은 프로그램 작업에 필요한 파일들을 보관하고, 변경 이력을 관리하는 공간입니다.

단순히 파일을 올려두는 저장소가 아니라, 어떤 파일이 언제 어떻게 바뀌었는지 확인할 수 있도록 작업 이력을 함께 관리합니다.

업무 자동화 프로그램을 만들 때도 코드 파일, 설명 문서, 설정 파일처럼 여러 파일이 함께 사용됩니다. GitHub은 이러한 파일들을 한곳에 보관하고, 작업 과정에서 생기는 변경 내용을 확인할 수 있게 해줍니다.

비유
회사 공유 폴더와 무엇이 다른가
공유 폴더는 최신 파일만 남지만, GitHub은 예전 버전까지 전부 기록해둡니다
사내 공유 폴더에 파일을 올리면 보통 최신 버전만 남고, 누가 언제 무엇을 바꿨는지는 파일명(예: "최종_v3")으로 겨우 짐작하게 됩니다. GitHub은 이 문제를 해결합니다. 파일이 바뀔 때마다 그 이력을 통째로 기록해서, 언제든 이전 버전으로 돌아가거나 "누가 무엇을 바꿨는지" 정확히 확인할 수 있습니다.

여기서 처음 들으면 헷갈리는 두 단어가 있습니다. Git과 GitHub입니다.
이름이 비슷하지만 역할은 다릅니다.

Git
내 PC 안에서 동작하는
변경 이력 기록 도구
  • 내 컴퓨터 안에서 동작
  • 파일이 바뀔 때마다 저장 지점을 남김
  • 인터넷 연결 없이도 사용 가능
  • 비유: 게임의 "저장하기" 기능
GitHub
Git으로 남긴 기록을
온라인에 올려 공유하는 공간
  • 인터넷에 있는 온라인 저장 공간
  • Git으로 기록한 이력을 온라인에 보관
  • 다른 사람과 함께 보고 이어받을 수 있음
  • 비유: 저장한 게임 파일을 올려두는 클라우드
SECTION 02

GitHub은 왜 필요한가

앞에서 본 것처럼 GitHub은 파일과 변경 이력을 관리하는 공간입니다. 특히 AI Agent와 함께 작업할 때는 GitHub의 역할이 더 중요해집니다.

AI Agent는 작업 폴더 안의 여러 파일을 한 번에 확인하고 수정할 수 있습니다. 작업 속도는 빨라지지만,
동시에 무엇이 어떻게 바뀌었는지 확인하는 기준도 필요합니다.

GitHub을 사용하면 AI Agent가 만든 결과물을 안전하게 남기고, 필요하면 이전 상태를 확인할 수 있습니다.
또한 작업 이력이 남아 있으면, 나중에 팀에서 결과물을 확인하고 이어받기 쉽습니다.

구분 역할
AI 개발 도구 AI Agent와 함께 실제 작업을 진행하는 프로그램(예: Claude Desktop)
작업 폴더 AI Agent가 파일을 열고 수정하는 내 PC의 실제 작업 공간
GitHub 작업 결과, 변경 이력, 프로젝트 기준 문서를 관리하는 온라인 저장 공간
왜 필요한가요?

AI Agent가 여러 파일을 수정한 뒤에는 결과만 보는 것이 아니라, 어떤 파일이 바뀌었는지 확인할 수 있어야 합니다. GitHub을 사용하면 변경 내용을 기록하고, 필요할 때 이전 상태를 참고할 수 있습니다.

SECTION 03

Repository와 Clone 이해하기

GitHub을 사용할 때 지금 단계에서 먼저 알아야 할 개념은 Repository(프로젝트 저장소)와 Clone
(내 PC로 복사)입니다.

앞서 GitHub을 "파일과 변경 이력을 보관하는 온라인 공간"이라고 했는데, 그 공간 하나에는 프로젝트별로 나뉜 저장소가 여러 개 들어 있습니다.
Repository는 그 안에 있는 프로젝트 저장소 하나하나를 가리키는 말입니다.
GitHub이 건물이라면, Repository는 그 건물 안에 있는 사무실 하나하나와 같습니다.
사무실(Repository) 안에는 코드 파일, 설명 문서, 설정 파일, 폴더 구조가 함께 들어가며, 파일이 어떻게 변경되었는지도 함께 관리됩니다.

Clone은 GitHub에 있는 Repository를 내 PC 작업 폴더로 복사해 가져오는 과정입니다.
GitHub 웹사이트에서 직접 작업한다는 뜻이 아니라, 온라인에 있는 저장소를 내 PC에 복사해 작업할 준비를 하는 단계입니다.

AI Agent는 GitHub 웹사이트 위에서 바로 일하는 것이 아니라, 내 PC로 복사(Clone)된 작업 폴더를 기준으로 파일을 확인하고 수정합니다. 그래서 처음 시작할 때는 GitHub의 저장소(Repository)를 복제(Clone)하고, 이후 Claude Desktop과 같은 AI 개발 도구에서 해당 작업 폴더를 열어 작업합니다.

개념 쉽게 말하면 역할
GitHub 건물 전체 여러 프로젝트 저장소(Repository)를 담고 있는 온라인 공간
Repository 건물 안의 사무실 하나 프로그램 작업에 필요한 파일과 변경 이력을 모아두는 프로젝트 단위 저장소
Clone 내 PC로 가져오기 GitHub의 Repository를 내 PC 작업 폴더로 복사하는 과정
작업 폴더 실제 작업 공간 AI Agent가 파일을 보고 수정하는 내 PC의 폴더
비유
도서관 원본 vs 내 책상 위의 복사본
Repository는 도서관에 있는 원본, Clone은 그 책을 복사해 내 책상에 올려둔 것
Repository는 도서관 서가에 꽂혀 있는 원본과 비슷합니다. 그 자체로는 작업할 수 없고, Clone으로 내 PC(내 책상)에 복사해와야 실제로 펼쳐보고 수정할 수 있습니다. 한 번 Clone하면 그 폴더는 내 PC에 그대로 남기 때문에, 다음에 작업할 때는 다시 Clone할 필요 없이 그 폴더를 열기만 하면 됩니다.
SECTION 04

사내 템플릿 Repository로 시작하는 이유

처음부터 빈 폴더에서 개발을 시작하면 폴더 구조, README* 작성 방식, 기본 설정을 모두 직접 정해야 합니다. 사람마다 시작 방식이 달라지면 나중에 결과물을 확인하고 이어받기도 어려워집니다.

사내 템플릿 Repository는 이런 기본 틀을 미리 갖춰둔 양식입니다. 이 템플릿을 복사해서 시작하면 DSR 기준에 맞는 구조 안에서 프로젝트 내용을 채워 넣는 방식으로 작업할 수 있습니다.

또한 사내 템플릿에는 작업 안내 문서와 보안 관련 문서가 함께 포함되어 있습니다. AI Agent는 작업 폴더 안의 파일을 기준으로 작업하기 때문에, 템플릿으로 만든 저장소(Repository)를 복제(Clone)하면 AI Agent가 사내 기준과 보안 관련 안내를 참고할 수 있는 상태에서 작업을 시작할 수 있습니다.

회사 공문을 매번 빈 문서에서 새로 만들지 않고, 정해진 공문 양식을 복사해 작성하는 것과 같습니다.

사내 템플릿은 출발점이고, Clone은 그 출발점을 내 PC 작업 폴더로 가져오는 과정입니다. 이렇게 시작하면 사람과 AI Agent가 같은 기준 문서를 보면서 작업할 수 있습니다.

SECTION 05

DSR-family 조직과 개인 GitHub의 차이

GitHub 계정은 크게 두 가지 기준으로 나뉩니다. 누구나 만들 수 있는 개인 계정과, 회사 기준으로 관리되는 DSR-family 조직 계정입니다.

개인 계정에 만든 Repository(저장소)는 개인이 관리합니다. 반면 DSR-family 조직 아래 만든 Repository(저장소)는 회사 차원에서 접근 권한을 관리하고, 담당자가 바뀌어도 조직에 그대로 남아 다른 팀원이 이어받을 수 있습니다.

구분 개인 계정 DSR-family 조직
소유·관리 개인이 직접 소유하고 관리 회사 차원에서 관리
접근 권한 기본적으로 본인만 접근 담당자·팀 기준으로 권한 설정
담당자 변경 시 계정 소유자가 바뀌면 인수인계 어려움 조직에 남아 다른 팀원이 이어받기 쉬움
주로 쓰는 용도 개인 학습, 사이드 프로젝트 회사 업무 관련 Repository
SECTION 06

Public과 Private의 차이

Repository를 만들 때는 Public과 Private 중 하나를 선택하게 됩니다. 이 설정은 나중에 바꿀 수도 있지만, 처음부터 올바르게 선택하는 것이 안전합니다.

구분 의미 접근 범위
Public 누구나 볼 수 있도록 공개 제한 없이 누구나
Private 지정된 인원만 접근 가능 Repository에 초대된 사람만
주의

회사 업무와 관련된 코드, 문서, 설정 파일에는 사내 정보가 포함될 수 있습니다. 이런 Repository는 기본적으로 Private로 만들어야 하며, Public으로 설정하면 사내 정보가 외부에 그대로 노출될 수 있습니다.

SECTION 03

Claude Code

Claude Code는 앞서 Chapter 02에서 살펴본 AI Agent에 해당합니다.

3단계 실습은 Claude Desktop의 여러 기능 중 Claude Code를 활용해 업무자동화 프로그램을 만드는 과정입니다.

사용자는 만들고 싶은 기능과 업무 기준을 설명하고, Claude Code는 필요한 코드 파일을 작성하고 오류를 수정하며 프로그램을 개발합니다. 즉, Claude Code는 단순히 코드만 보여주는 기능이 아니라, 작업 폴더 안의 파일을 기준으로 개발 흐름을 이어가는 기능으로 이해하면 됩니다.
